# ADR-0002: ファジーマッチングアルゴリズム選定

## ステータス

Accepted

## コンテキスト

POS Voice Concierge の音声認識結果（Whisper base モデル出力）から商品マスタを照合するファジーマッチングエンジンが必要である。以下の要件を満たすアルゴリズムを選定する:

1. 日本語文字列に対して安定したスコアリングが可能
2. 表記ゆれ（カタカナ/ひらがな混在、中黒の有無、長音の揺れ、略称）を吸収
3. 1000 商品マスタに対して < 100ms のレイテンシ
4. Python エコシステムで成熟したライブラリがある

### 候補

| ライブラリ         | アルゴリズム                    | 日本語対応 | 速度        | ライセンス |
|--------------------|---------------------------------|-----------|------------|-----------|
| rapidfuzz          | 複数（WRatio, ratio, etc.）      | 良好       | C++ 実装で高速 | MIT       |
| python-Levenshtein | レーベンシュタイン距離           | 普通       | 中速        | GPL       |
| fuzzywuzzy         | WRatio（Python 実装）            | 良好       | 低速        | MIT       |
| difflib            | SequenceMatcher                 | 普通       | 低速        | 標準ライブラリ |
| Elasticsearch      | n-gram / 形態素解析             | 最良       | 要外部サービス | Apache 2.0 |

## 決定

**rapidfuzz の WRatio スコアラーを使用する。**

### 選定理由

1. **WRatio が日本語に最適**: WRatio は内部で ratio, partial_ratio, token_sort_ratio, token_set_ratio を組み合わせ、最も高いスコアを返す。これにより以下の表記ゆれを自然に吸収する:
   - 中黒の有無: 「コカ・コーラ」vs「コカコーラ」→ partial_ratio で高スコア
   - 語順の違い: 「ポテトチップス うすしお カルビー」vs「カルビーポテトチップス うすしお」→ token_sort_ratio で吸収
   - 部分一致: 「コーラ」vs「コカ・コーラ 500ml」→ partial_ratio で吸収

2. **C++ 実装による高速性**: rapidfuzz は C++ で実装されており、Python の fuzzywuzzy と比較して 10-100 倍高速。1000 商品マスタに対して 100ms 以内のレイテンシ要件を満たす。

3. **process.extract の効率性**: 全候補に対する一括検索を最適化された C++ コードで実行。個別に比較するよりも大幅に高速。

4. **score_cutoff の早期打ち切り**: 閾値以下のスコアを早期に除外することで、不要な計算を省略。

5. **GPL フリー**: python-Levenshtein は GPL のため、MIT ライセンスの本プロジェクトでは採用不可。rapidfuzz は MIT ライセンス。

### 閾値設計

- デフォルト閾値: **80%**
- 根拠: Whisper base モデルの日本語認識精度を考慮し、誤認識を含む入力でも正しい商品にマッチする下限値。60% 未満では誤マッチが増加し、90% 以上では表記ゆれを吸収できない。
- カスタマイズ可能: `FuzzyMatcher(threshold=...)` で店舗ごとに調整可能。

### 表記ゆれ辞書の併用

WRatio だけでは吸収しきれない表記ゆれ（例: 「ポテチ」→「カルビーポテトチップス うすしお」）は、エイリアス辞書で補完する設計とした。

- **手動登録**: 事前に既知の略称・通称を登録
- **自動学習**: ユーザーの手動修正をトリガーにエイリアスを自動登録
- **エクスポート/インポート**: JSON 形式で辞書の持ち出し・共有が可能

### 不採用理由

- **Elasticsearch**: 形態素解析による日本語検索は最も精度が高いが、外部サービスの運用コストが POS 端末向けシステムに見合わない。将来的に商品数が 10,000 を超える場合に再検討。
- **difflib**: 標準ライブラリで依存が少ないが、速度が遅く、1000 商品で 100ms 要件を満たせない。
- **fuzzywuzzy**: WRatio のオリジナル実装だが、Pure Python のため速度が不十分。rapidfuzz はこれの C++ 再実装。

## 影響

- voice-engine の依存に `rapidfuzz>=3.6` を追加（既に追加済み）
- `FuzzyMatcher` クラスに `process.extract` + `fuzz.WRatio` を使用
- `utils.default_process` で前処理（大文字/小文字正規化、空白トリム）を適用
- 商品数が 10,000 を超える場合は、キャッシュまたは n-gram インデックスの導入を検討
- 将来的に MeCab 等の形態素解析を前段に追加する可能性あり（別 ADR で決定）
